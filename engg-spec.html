Sandpaper

Engineering spec — P0, the spine.

A local Node.js bridge — six files, zero runtime dependencies — that lets you refine a Claude-generated HTML document in the browser without leaving the page. Point at an element, say what to change, and the file updates in place. P0 delivers the spine: serve + inject, Claude round-trip, live reload, status chip, click-to-scope.

The invariant

The HTML file on disk is the document. The server serves it; Claude edits it; the watcher reloads it. Nothing in between — no server-side document model, no HTML assembled from stdout, no diffing. The page always reflects exactly what is on disk.

01
Overview

What P0 is, and the one rule that governs it

Run it with node bin/cli.js <document.html>. The server binds to 127.0.0.1:4848 (override with SANDPAPER_PORT). The document appears exactly as it would opened from disk — plus a toolbar overlay in the lower-right.

Single source of truth. Every read and every write — whether you edit the file in a text editor or Claude applies an Edit tool call — targets the same file. The bridge is a dumb pipe: Claude edits the file with its own tools; fs.watch detects the change and tells the browser to reload. That invariant rules out "apply the diff from stdout", streaming partial HTML, or any server-side document model.

Scope. P0 = serve+inject, claude round-trip, live reload, status chip, click-to-scope. P1 adds direct manipulation (drag/delete → file write) and screenshot paste. P2+ adds durable data-cid anchoring, branching, and a long-lived watch session.

Node ≥18 built-ins only: http, child_process, fs, path, url. No npm install. ESM.

03
The bridge

Flags, the stream → status map, session persistence

The exact invocation, spawned with cwd = docDir so --resume's directory-scoped session lookup stays reliable:

claude -p <prompt> \
  --output-format stream-json --verbose --include-partial-messages \
  --permission-mode acceptEdits \
  --allowedTools Read,Edit,Write,MultiEdit \
  --append-system-prompt <CONTRACT> \
  [--resume <sessionId>]

Verified against the docs: --allowedTools as a single comma-separated arg is valid; --append-system-prompt appends to (doesn't replace) the default system prompt; acceptEdits auto-approves file edits without granting shell access; --verbose is recommended, not strictly required. Tools are restricted to file access only — no Bash, no web, no MCP.

Event → status mapping

Session. The session id is captured from the init event and written to <docDir>/.sandpaper/session.json; every later turn loads it and passes --resume. Sessions are per-document-folder. Even if it's lost, the file is the memory — a fresh session rebuilds context by reading the document.

mapEvent() and getSessionId() are pure functions, unit-tested against a recorded 8-event fixture. The waiting state exists in the chip but isn't emitted yet — reserved for an interactive-permission flow.
04
The page protocol

HTTP surface, the turn, click-to-scope

method · pathpurpose
GET /Serve the document with the toolbar injected before </body> (verbatim read; the file is never modified to serve it).
GET /__sandpaper/eventsSSE. Sends an initial idle frame, then status objects {state,label,detail?} and reload signals {type:'reload'}.
POST /__sandpaper/turnBody JSON {prompt, cid?, selector?, snippet?} (≤1 MB). Builds the prompt, runs the turn, returns 202 — or 409 if a turn is already in progress.
GET /__sandpaper/toolbar.{js,css}Serve the overlay assets from the package's own public/.
GET /<path>Serve files under docDir (images, pasted screenshots), path-traversal guarded.

Anatomy of one turn

  1. Optionally click ⌖ and click an element. The picker prefers its nearest data-cid; else it builds a CSS path (≤6 levels, :nth-of-type) + an 80-char text snippet.
  2. Type an instruction and submit. The toolbar ignores the submit if a turn is already running (busy guard).
  3. Chip → Sending…; the toolbar POSTs {prompt, cid?, selector?, snippet?} to /turn.
  4. Server builds a scoped prompt ("edit ONLY the element with data-cid=X …") and calls runTurn(); responds 202 immediately.
  5. claude.js loads the session id, emits init, and spawns claude -p from docDir.
  6. stdout NDJSON → mapEvent() → broadcast() → SSE → chip moves thinking → editing.
  7. Claude's Edit tool writes the change to the file on disk — the server is not involved in the write.
  8. fs.watch fires → 120 ms debounce → broadcast({type:'reload'}).
  9. The result event → chip Saved to <doc> · undo? with cost.
  10. Toolbar saves scroll, reloads; the page re-serves from disk with the toolbar re-injected.
Status and reload share one SSE channel, distinguished by a state field vs a type:'reload' field. The 202 is fire-and-forget; the chip is driven entirely by SSE.
05
Security model

A local-only tool, scoped tight

controldetail
Loopback bindServer listens on 127.0.0.1 only — never 0.0.0.0. Not exposed to the network.
Tool restriction--allowedTools Read,Edit,Write,MultiEdit — no Bash, no web fetch, no MCP. Claude can't run shell or open connections during a turn.
Permission modeacceptEdits authorizes file writes in docDir without prompts; grants no broader access.
Path-traversal guardnormalize(join(docDir,path)).startsWith(docDir+sep) → 404 on escape. Malformed percent-escapes return 400 (guarded decode).
Body cap/turn body capped at 1 MB; the socket is destroyed past the limit.

Accepted residual: no request authentication — any local process running as you can POST a turn. Acceptable for a local-only tool whose worst case is still bounded by the file-only tool restriction.

Threat model is "your own machine." The interesting hardening later is scoping which directories a turn may touch — today it's the doc's folder.

07
Found & fixed

What the adversarial review caught

A code-review agent read the real source; a Claude-Code-mechanics agent re-verified every flag against the docs. Eight confirmed bugs — all fixed this pass. One item stays genuinely open.

Critical
claude.js · close handler — a spawn failure (ENOENT, auth) fired error then close(null); if (code && code!==0) is falsy on null, so it broadcast idle and hid every error. Fixed with an errored flag that short-circuits close.
High
server.js · fs.watch — fname can be null (Linux/inotify, macOS under load); the old guard dropped the reload. Fixed — null is treated as a match.
High
claude.js · stdout — no end listener; a final line without a trailing newline lost the result event (cost + "Saved"). Fixed — an end-flush drains the buffer.
High
server.js + toolbar.js — no concurrency guard; two turns could race-write the file. Fixed — activeTurn gate + 409 + a toolbar busy-check.
Medium
server.js · broadcast() — res.write() on a destroyed socket could crash the process mid-turn. Fixed — try/catch + evict.
Medium
server.js · pathname — the reviewer flagged a double-decode; on inspection the real risk was decodeURIComponent throwing on malformed input. Fixed — guarded decode → 400 (kept decoding so filenames with spaces still work).
Medium
claude.js · spawn — default stdio left stdin an open pipe (potential hang). Fixed — stdio:['ignore','pipe','pipe'].
Low
toolbar.js · scroll — restore ran before layout settled, clamping scroll on image-heavy pages. Fixed — deferred to the load event.
Open ?
claude.js · result event shape — the exact result event fields aren't documented in the CLI reference (inferred from the Agent SDK + the fixture). To verify against a live turn and update the fixture if needed.
The reviewer's fix #6 was itself slightly wrong (it assumed url.pathname is fully decoded — it isn't). Verifying findings, not blindly applying them, is the point.
08
Testing & how to run

Run it, test it

# run the bridge (point it at any HTML document)
node bin/cli.js sandpaper.html       # → http://127.0.0.1:4848

# parser unit test — no live model call, <100ms
npm test                             # init → thinking → editing → done ✓

Covered: the stream-json → status mapping and session capture (pure functions, recorded fixture). Live-smoke-tested: serve+inject (200), toolbar assets (200), SSE idle frame, path-traversal (404), malformed-percent (400).

Not yet covered: HTTP routing, the watch/reload integration, SSE set management, toolbar behavior, and the error paths. To add: a unit test for the errored flag (mock error then close(null), assert no idle), an end-flush test, and a 409 concurrency test.

The one live step: firing a real claude -p turn from the page. It was built with doc-verified flags but not auto-run (to avoid nesting a Claude inside a Claude) — it's the first thing to exercise, and the moment to confirm the open result-shape question.

Fastest manual check: node bin/cli.js sandpaper.html, open the URL, click ⌖, click the hero subtitle, type "tighten this", watch the chip and the change.
