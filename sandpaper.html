01
The problem

The medium got richer. The loop didn’t.

The loop today

  • You brief Claude; it writes a spec — first as markdown. Readable, but flat: no diagrams, no mockups, hard to feel the design.
  • So you moved to HTML. Now the document shows the idea — illustrations, tables, real mockups. Far better at expressing the problem.
  • But to change anything you leave the document, alt-tab to the terminal, and describe in words the thing you were just looking at.
  • Claude often regenerates a whole new HTML; you re-read it, find the next thing, alt-tab again. Linear, lossy, one thread.

The loop with Sandpaper

  • Read the rich document exactly as you do now.
  • Point at the paragraph, table, or mockup and say the change — or edit it directly, in place.
  • It updates live. Direct text, move, delete, and undo ship today; richer non-linear tools remain on the roadmap.
  • It stays one durable file — the canonical spec your developers build from.
The friction was never the editing — Claude is fast. It’s the surface mismatch: you see the work as a rich document, but you steer it as text in another window.

Architecture

How it works

One idea carries the whole design: the document on disk is the single source of truth — and you, Claude Code, and Codex read and write that same file.

The bridge invokes either claude -p --resume or controlled codex exec resume from the document’s folder and normalizes its event stream. The provider edits the HTML; the watcher refreshes the page. Your direct edits — drag, delete, reorder — write to that same file with no AI call at all: instant, free, exact.

Because the document is the source of truth — not the chat — context survives everything. If a session drops, a fresh agent reconstructs the full picture by reading the file and shared brain. Page/provider-scoped resumable sessions accelerate work, but provider sessions are not canonical project memory. And the same file is what your developers open next — Sandpaper doesn’t need to be running for the spec to be real.

Today the supported editing surface is HTML served from a file or repository directory. The toolbar gives both providers first-class explicit selection with no silent fallback and no hidden context handoff. A tab-local selection can supersede the launch choice; only Make default writes the local manifest.

Codex reuses saved codex login authentication in a controlled embedded runtime; interactive $sandpaper continues in the user's normal Codex environment.

Terminal errors remain visible, rejected requests restore the composer, and undo is shown only for a real unconsumed snapshot. The server, not a requested tool call, decides what saved.
05
For sign-off

The one-page product spec

Problem

Your medium for thinking with Claude got richer — you moved from flat markdown to HTML so specs could show diagrams, tables, and mockups, not just describe them. But the collaboration loop didn’t keep up. You still read the document in one window and steer it from another, re-describing in words a change you’re looking right at. For design and spec work that round-trip is the bottleneck: lossy, linear, flow-breaking — and the document you’re shaping isn’t throwaway chat, it’s the spec your build depends on.

Hypothesis

If people can read the rich document and change it in place — pointing, editing, and talking to it while the explicitly selected provider rewrites the file as the single source of truth and shows its live status — then design collaboration collapses from a terminal monologue into a fast, non-linear conversation on the page, and the artifact stays canonical enough to drive what gets built.

Experience

You open the spec your agent made — diagrams, tables, a mockup. You hover; a section highlights. You click, say “make this a three-column comparison and warm the intro,” and hit enter. The chip flips to Thinking, then “Editing spec.html,” the page updates in place, and the chip settles on Saved with Undo only when bytes changed, otherwise Replied. Hands edits, moves, deletes, and restores blocks directly. Branching and screenshot paste remain roadmap work; the file you finish with is still the one your team builds from.

Acceptance criteria
  • I can select any visible element and Sandpaper resolves it to the exact spot in the HTML file.
  • When I describe a change, the edit lands in the file on disk and updates the document in under ~10 seconds.
  • A direct-manipulation gesture (reorder, delete a block, edit text) is written to the file immediately, with no AI call and no flicker.
  • Planned: paste a screenshot and let Claude reference it as real visual context.
  • The status chip reflects server-confirmed thinking, editing, error, Saved, or Replied state and always recovers after rejection.
  • Claude Code and Codex are explicit toolbar choices; switching transfers no transcript or hidden context and never silently falls back.
  • The finished document is valid, shareable HTML that opens and reads correctly even with Sandpaper not running.
  • Edge: malformed, network, authentication, runner, or conflict failures leave the composer usable and show the real error.
  • Planned: if Claude regenerates the document, open annotations re-anchor or report that they could not.
Success metrics
Leading
Median time from point → change-visible-in-document is under 30 seconds across a session.
Lagging
60%+ of refinements happen on the page instead of round-tripping to the terminal, within two weeks of use.
Guardrail
Under 2% of Sandpaper edits are undone or reverted on the next turn — the trust-and-quality check.

07
Honesty

Open questions & risks

  • ContinuityLargely dissolved. The file and brain are durable memory; each page/provider resume ID and browser transcript stays separate. New session clears only that scope after server success.
  • AnchoringKeeping annotations attached when Claude regenerates the HTML. Mitigation: a CLAUDE.md contract to never strip data-cid, plus text/selector fallback.
  • Latency~1–2s cold start per semantic turn. Mitigation: optimistic local edits feel instant; a P4 watch session removes the cold start.
  • SecurityProvider processes have directory-level write access on localhost, not OS-level single-file isolation. Mitigation: loopback bind, per-process token, same-origin checks, body caps, controlled Codex flags, server byte/hash authority, and best-effort external-path warnings. External changes cannot be verified or undone by Sandpaper.
  • ScopeThis is a refinement loop for documents, not an IDE or a Figma. The discipline is knowing where Sandpaper stops.
  • BillingSubscription reuse holds today (verified), but the paused change could return. Mitigation: keep the brain swappable.
  • SubtractiveThe name leans toward “smooth / remove,” but the tool also adds richness. Mitigation: keep one-click expand/branch first-class so Sandpaper earns the generative half, not just polish.
Claude Code and Codex can both drive the page. Provider selection is explicit, unavailable states are actionable, and there is no silent fallback.
08
Decided

On the name

Sandpaper. An everyday word with instant recall that rolls off the tongue in the “-paper” family (newspaper, Pied Piper) — and just weird-and-fresh enough to be ownable. More than that, it names the actual job: you take Claude’s rough draft and sand it into a finished, build-ready spec. Tactile, honest craft — the opposite of small.

The one thing we’ll stay honest about: sandpaper is subtractive by nature, so the product has to keep earning the “make it richer,” not only the “smooth it.” That’s a feature constraint, not a name problem.

Considered & set aside — Tweaq (too small) · Illume / Gilden (too rare) · Vivid (warm, but less ours). Domain: sandpaper.app is taken, so likely getsandpaper / sandpaper.dev.

You reached for a tactile craft metaphor twice (Claywork, then Sandpaper) — the warm-paper, clay palette of this very page was already living in that world.
