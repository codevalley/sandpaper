Product · what & why

What we're building, and why it earns its place

Sandpaper turns a Claude-generated HTML draft into a surface you refine in place — point, say, or edit directly, no terminal round-trip. The pitch, the interaction modes, and the product calls live in the records below; this page routes you in.

Why it exists · the bet

The medium got richer; the loop didn't

Claude Code writes your specs as rich HTML — diagrams, tables, mockups — because HTML shows the idea instead of merely describing it. But to change one line you alt-tab to the terminal and re-describe the thing you were just looking at. Claude regenerates a whole new document; you re-read it, find the next thing, alt-tab again. Linear, lossy, one thread. The friction was never the editing — it's the surface mismatch. the loop, side by side →

The bet: if you can read the rich document and change it in place — while Claude rewrites the file as the single source of truth and shows its live status — then design collaboration collapses from a terminal monologue into a fast, non-linear conversation on the page, and the artifact stays canonical enough to drive what gets built. It runs on the Claude subscription you already pay for — claude -p still draws from the plan, verified. spec §01 →   why claude -p →

The June-15 billing change that could split the credit pool is paused, not gone — so the brain is kept swappable as insurance. the billing watch →

What it is

Refine the rendered page, in place

Sandpaper is a local tool that turns a Claude-generated HTML document into a surface you refine in the browser. A small Node bridge (a few hundred lines, zero runtime deps) drives claude -p in stream-json; when Claude edits the file, a watcher live-reloads the page, and a calm status chip says what it's doing — thinking · editing · done. No terminal round-trip, no re-describing. how it's built →   the status chip §04 →

One invariant carries the whole design: the document on disk is the single source of truth. Both you (direct edits) and Claude (Edit/Write) mutate the same file; the page reflects disk, never ahead of it. That rules out a server-side document model, HTML-from-stdout, and diffing — the bridge stays a dumb pipe, never screen-scraping a terminal. And the file you finish with is valid, shareable HTML that opens correctly with Sandpaper not running — the spec your team builds from. the SSOT call →   the component map →   why not scrape →

Input · the ways in

Point to scope, then choose a hand

Every interaction starts the same way: hover, then click an element or lasso a region, and Sandpaper resolves it to the exact spot in the HTML file. With the target scoped, there are three ways in.

Way inWhat it doesEdits via
SandScoped AI edit — say the change (“tighten this, use our clay”) or one-click refine (expand · simplify · shift tone). Claude edits just that block; the page live-reloads.Claude
✎ HandsDirect manipulation — drag to reorder, delete, or edit text in place. Written straight to the file: no AI call, zero latency.you
⇥ SlingCopies a terminal-ready instruction for cross-cutting / multi-file work — you hand it across to the terminal that's already good at it.terminal

Sand and Hands are the two halves of one invariant — an AI file write and a human file write to the same source of truth. Sling is the deliberate seam: the page is the eyes, deep steering stays in the terminal. Point-and-say Sand shipped with the spine; Hands is in the build now. the modes §03 →   modes & promise →   Hands on the board →

Path · where it's going

Thin spine first, then the hard modes

The roadmap climbs a thin spine before it widens. P0 the spine — live status chip + point-and-say on one HTML doc — is shipped. P1 the two hard modes — direct manipulation and paste-a-screenshot — is in flight. P2–P4 — durable data-cid anchoring, branch & compare, and a long-lived watch session — are planned. The rungs are canonical in the spec; the live state of each is the project board, not this page. roadmap §06 →   the plan board →

The one rung that's also a blocker: keeping annotations attached when Claude regenerates the whole HTML. Tracked as an open question and a named spec risk. the anchoring question →   risks §07 →

The wedge · why it earns its place

The page is state made visible; the terminal stays the mouth

Sandpaper made one hard pivot: the browser is the eyes, not the chat. An in-browser chatbox is just a worse terminal, so the wedge isn't chat parity — it's the two things a terminal can't do. The page makes state visible: the rendered artifact, the live status chip, even this project brain. And it makes the artifact touchable: point, Sand, Hands. Deep, cross-cutting steering stays where it's already good — the terminal, the mouth — reached by Sling. the pivot →

The discipline is knowing where it stops: a refinement loop for documents, not an IDE, not a Figma. The proof is in the dogfood — this very brain is served by Sandpaper, where Sand refines one page in place and Sling hands cross-cutting work over.
