Engineering · how it's built

The bridge, the contracts, the anchoring

How Sandpaper works under the glass: the dumb-pipe bridge that drives claude -p in stream-json, the marker contract the brain stamps against, and the open anchoring problem. The records below are canonical — this page only routes you in.

01 · The invariant

The file on disk is the document

One rule governs everything else: the HTML file on disk is the document. The server serves it verbatim, Claude edits it with its own tools, the watcher reloads it — and the page always reflects exactly what is on disk, never a step ahead. No server-side document model, no HTML assembled from stdout, no diff-and-patch dance. engg →

Because both you (typing in an editor) and Claude (an Edit call) mutate the same bytes, the two sides converge on one artifact instead of drifting. It is also why memory is cheap: a fresh session rebuilds its whole context by reading the file, so the session id is an accelerator, not a dependency. decision → · the file is the memory →

Everything below is downstream of this one invariant — the bridge, the contract, and the Hands all exist to keep disk authoritative.

02 · The bridge

A dumb pipe over claude -p

The bridge is a dumb pipe, a few hundred lines: it spawns Claude, streams status to the page, and stays out of the way of the write. It drives the model headlessly with claude -p --output-format stream-json (plus --include-partial-messages, --permission-mode acceptEdits, --allowedTools Read,Edit,Write,MultiEdit, and a --append-system-prompt contract), spawned with cwd = docDir so --resume's folder-scoped session lookup stays reliable. Never tmux, never screen-scraping a TUI. engg §03 → · why claude -p →

It is the same Claude as the interactive tool — only its output is a structured event stream. mapEvent() turns each stream-json frame into a chip state — init → thinking → editing → done — riding the same SSE channel as the reload signal. event map →

Two gotchas here are load-bearing. Scrub the child env before spawning: an exported ANTHROPIC_API_KEY silently bills the API even on a plan, so the bridge drops it (and the CLAUDECODE markers) to keep billing the subscription. And the watcher fires loose: fs.watch(doc) → a 120 ms debounce → broadcast({type:'reload'}) → SSE → the page reloads from disk and restores scroll — but the watch event can carry a null filename, so treat null as a match or you silently drop reloads. env scrub → · null fname →

03 · The contract

How the brain stays current

This brain is not free-form prose — it is regenerated against a fixed contract so an agent can keep it true without drift. Live state sits in marker-delimited regions (the <!-- BRAIN:NOW --> block and a brain-state JSON digest on the cover) that get overwritten whole, never prose-edited in place. the contract →

Every durable fact is an .entry: a block carrying data-cid (mirrored to its id), data-kind, data-status, data-date, and at least one data-ref into a canonical anchor — plus an optional data-lens for the three-lens axis. Status is a lifecycle flipped on the block, never a rewrite — a reversal is a new entry that supersedes the old. The governing rule is link, never copy: the living books point at sandpaper.html / engg-spec.html rather than restating them. the ledger → · the map →

Skin is one file. All design tokens live in theme.css; brain.css only @imports and consumes them via var(), so re-skinning a whole Sandpaper is a one-file edit (the injected toolbar keeps its own scoped copy so it renders on any host page). The catch that bit us: @import loads first, so a leftover :root in the consumer silently wins — drop it entirely. decision → · cascade gotcha →

04 · The Hands

Direct manipulation, no model in the loop

Not every edit needs Claude. The Hands are the page acting on the document directly — edit text in place, drag to reorder, delete a block — each written straight to the file with no AI call. Editing text in place already ships; drag-and-delete is next. the plan → · engg §09 →

The mechanism falls out of the invariant: the page knows the element's data-cid, so the write locates that one element in the file and replaces only its bytes, leaving everything around it untouched — a byte-preserving slice keyed on the anchor, not a re-serialized DOM or a whole-file rewrite. A pre-turn snapshot backs the undo? the chip already promises. make undo real →

This is the third writer on one file — your keystrokes, Claude's tools, and the page's own edits all targeting a single source of truth.

05 · Components & the open problem

A handful of files, and the anchor that has to survive

The whole bridge is a handful of files with one direction of dependency — cli → server → claude at its core, plus the on-page toolbar — and zero runtime dependencies (Node ≥18 built-ins, ESM, nothing to npm install). Roles and live build status live on the component map; the authoritative status stays in the engg spec. component map → · how the pieces fit →

The one genuinely open engineering problem is anchoring. data-cid is both the brain's link target and the unit of scoping, so when Claude regenerates a document wholesale and drops the cids, every open annotation and brain link dangles. The plan: a CLAUDE.md contract that forbids stripping cids, backed by a text-snippet / selector fallback. Until that lands, durable anchoring is the gating risk for P2. open question → · risks §07 →
