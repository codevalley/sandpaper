C.1 The invariant

The one rule

The HTML file on disk is the document. The server serves it, Claude edits it with its own tools, the watcher reloads it. No server-side document model, no HTML assembled from stdout, no diffing — the page always reflects exactly what is on disk. engg →

C.2 Data flow

browser · toolbar→ POST /turn →server→ spawn →claude -p→ Edit →doc on disk
fs.watch(doc) → broadcast reload → SSE → page reloads from disk

Outbound and inbound are separate paths over one local server. A turn writes the file; the watcher pushes the reload. Status rides the same SSE channel as a typed event stream. engg §02 →

C.4 Anatomy of one turn

  1. Click ⌖, click a section (it resolves the nearest data-cid), type a change.
  2. Toolbar POSTs {prompt, cid?}; the server builds a scoped prompt and spawns claude -p --resume from the doc's folder.
  3. stream-json events map to the status chip (thinking → editing) and to streamed reply / edit cards.
  4. Claude's Edit writes the change to the file — the server is not involved in the write.
  5. fs.watch fires → 120 ms debounce → one reload at turn-end; the page restores scroll.

C.6 Security

A local-only tool, scoped tight: bound to 127.0.0.1; --allowedTools Read,Edit,Write,MultiEdit (no Bash, web, or MCP); acceptEdits for in-folder writes only; path-traversal guard; 1 MB body cap. Accepted residual: no request auth on a single-user local server.
