C.1 The invariant

The one rule

The HTML file on disk is the document. The server serves it, Claude edits it with its own tools, direct operations splice it, and the watcher reloads it. Server hashes decide whether a turn really saved bytes; no stdout becomes document content. engg →

C.2 Data flow

browser · authenticated client→ POST /turn →server lifecycle→ runner →claude -p→ Edit →doc on disk
hash + snapshot + fs.watch → page/client-scoped SSE → reload from disk

Outbound and inbound are separate paths over one local server. A turn writes the file; the watcher pushes the reload. Status rides the same SSE channel as a typed event stream. engg §02 →

C.3 The modules

Six runtime modules plus tests, one server-owned lifecycle, and zero runtime dependencies. The runner seam is provider-neutral internally; production and the current UI are Claude-only.

C.4 Anatomy of one turn

  1. Click ⌖, click a section (it resolves the nearest data-cid), type a change.
  2. The client POSTs authenticated JSON; the server reserves the turn before reading the body and snapshots the page.
  3. The current Claude runner emits normalized stream frames; the server owns IDs, status, hashes and terminal state.
  4. Claude's Edit writes the file; the server compares bytes and reports Saved/undo only for a real change.
  5. fs.watch routes page-scoped reloads, preserving only the initiating tab's confirmed optimistic direct edit.

C.6 Security

A local-only tool, scoped tight: bound to 127.0.0.1; per-process token plus same-origin/loopback checks on mutations and SSE; shared path policy denies secrets, runtime state, traversal, and escaping symlinks while allowing ordinary repo files and .github; Claude gets file tools only; request bodies are capped.
