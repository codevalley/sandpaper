07-02 · board 008

The landing page is built (site/ — static, zero-build, deployable as-is). The real problem wasn't design, it was explanation: demos to friends kept failing because the pitch led with mechanism. So the copy was chosen by a judged pitch lab — five writers, five angles, three blind judges scoring 15-second comprehension cold.

Winner · “artifact” (84)“Your repo can explain itself.” Lead with the thing you GET — a website inside the repo — not the mechanism.
Grafted from runners-up“stranger in your own repo” · “a hook won't let the session end unstamped” · the README-rot beat

The signature: a self-stamping hero demo — a terminal card types a real command, and the mini brain beside it stamps: log row lands, NOW rewrites, plan ticks. It re-enacts w-0193, the turn this repo went public. Show the loop, don't describe it. Sections are graded like sandpaper — P80 (the problem, coarse) → P2000 (principles, polish).

Then a 4-critic panel (cold-viewer · art director · copy chief · a11y) reviewed screenshots + source: 30+ findings, all real ones fixed — including a caught hand-tinted stat (31/35 counted initiatives as tasks; truth is task-only) on a page that brags “never hand-tinted”, insider jargon in the demo, WCAG contrast failures, and a wordmark anchor that scrolled the wrong way. Deploy: sandpaper.sh ← site/, brain.sandpaper.sh ← brain/ (the resolver makes that work).

The owner's invariant — brain/ deploys as-is, no build step — survived an adversarial panel. Measured: 158 out-of-brain links, 73% anchored spec refs GitHub's blob view can't render. Write-time absolute links and the publish/export step both judged dead.

07-01 · 006The brain links outside itself — on purpose›

The out-links aren't sloppiness — they're the spine. Sandpaper's one rule is link, never copy: the brain is a thin index over canonical truth (the spec, the source), never a second copy of it. Copy the content in and it drifts the moment the code changes.

They only break when brain/ is served detached from its repo (a subfolder-only deploy). Partly superseded by board 007: the fix moved from a publish step to a runtime resolver — the owner pushed back on “publishable only after a build,” and the adversarial panel proved them right.

06-30 · 005Board-first — the canvas is where explanations live›

The decision: when my reply would be a substantial summary or explanation, the board is the reply — it lands here, and the terminal gets one line (📋 … → on the canvas). Not written twice.

We dropped the Stop-hook “nag”. A hook can't tell what's board-worthy — it only sees length, so it over- or under-fires, and it never boards anything itself; it just pokes me, and the call is still mine. So the fix puts that judgment in the natural flow instead of bolting on a dumb detector.

06-30 · 004A scrollable whiteboard — and boarding that doesn't rely on memory›

This is a board — the kind you said my summaries should be. Two things landed this turn.

1 · The whiteboard is real. The canvas is now a white container with its own scroll: a long board scrolls inside it, while the whole page — this box, Earlier, the doors, the footer — scrolls together as one. A scrollable whiteboard, not a trapped panel.

2 · Boarding no longer relies on me remembering. Laying the turn's summary here is now a step of the STAMP — the routine a Stop hook already enforces on every substantive turn. So in any repo, when Claude stamps the brain (and it's blocked until it does), it lays the current summary on this whiteboard too. That's the mechanism, not a promise.
