Gotchas & verdicts

So they're never re-learned. One claim, one why, a back-link to where it bit. · ▲ gotcha · ◆ verdict

▲ Gotcha · security

Everything read from the target repo is untrusted input — a cloned repo's package name can be x" onload="…. The scaffolder interpolated the name and remote URL straight into HTML attributes and a hand-built JSON digest; caught by the post-commit security review. Fix: esc() for every attribute/text interpolation, JSON.stringify for the digest. setup.js →

▲ Gotcha · security

A git remote URL can carry credentials — never write it verbatim into a page built to be published. repoSource() derived the resolver's base from git remote get-url origin; an origin like https://user:TOKEN@github.com/… would have baked the token into all 13 brain pages. Caught by the adversarial review pre-publish; the fix strips userinfo before the base is ever written. setup.js →

▲ Gotcha · resolver

A click-time href rewrite left in the DOM becomes DATA the moment anything saves the DOM. The toolbar's edit-in-place commits innerHTML to disk — and stopPropagation() doesn't stop other listeners on the same node, so the resolver still fired under the toolbar and its absolute URL could be silently persisted into the brain file. Fix: rewrite just-in-time, then restore the raw href one tick later (+ stay inert when the click was already claimed). brain.js →

▲ Gotcha · canvas

CSS viewport math can't know where an element starts — max-height:calc(100vh - 184px) assumed a fixed offset and the canvas sailed past the fold. The shell + NOW plate above the whiteboard vary in height, so the constant was wrong the day it was written. Fix: brain.js measures the real document offset and sets the cap (re-fit on resize + font load); the CSS keeps only a min(56vh,600px) JS-off fallback. brain.js →

▲ Gotcha · distribution

Take away the reference and prose alone won't hold the structure — the brain collapses to a single page. After init stopped copying the dev's brain (l-init-templates), it had the assets but had to invent the page structure from the spec — and produced a one-page site with #anchor nav. Prose repeatedly loses to a shipped skeleton. Fix: install-skill now scaffolds the full multi-page skeleton (cover + 3 lens pages + 3 books, shell nav wired to relative paths); the slash command FILLS it, never invents. Ship structure, let the agent fill content. setup.js →

▲ Gotcha · distribution

“Write from the shipped templates” without saying WHERE makes the agent hunt — and grab the dev's own repo. A fresh /sandpaper:init needed theme.css/brain.css/brain.js but they weren't local (install-skill only copied commands + hooks), so it scanned the filesystem and used the sibling productcraft brain as its reference — which won't exist on a distributed install. Fix: install-skill now scaffolds the package's assets locally, and the slash command is told to USE brain/assets/ and NEVER read templates from any other directory. (Same pass: the go-ahead gate + wizard now use the AskUserQuestion dialog, not plain prose.) the command →

▲ Gotcha · theming

An injected widget that ships its own token copy won't re-skin with the host. The toolbar carries a private #sp-panel --sp-* palette so it stands alone on any page — so a theme.css re-skin reached the brain but not the floating toolbar. Fix: read the host's :root tokens at runtime (getComputedStyle) and override the copy when present; the shipped values become a fallback for un-themed pages. The toolbar re-runs each live-reload, so the sync is automatic. engineering →

▲ Gotcha · theming

Adopting a host's tokens blindly can break the chrome's own contrast contract. The toolbar pairs light text on its dark --sp-ink head; a theme whose --ink is a light/mid tone (fine as body text, wrong as a fill) washed the head out — the tinysync first-run tour. Fix: a luminance/polarity check — adopt the neutral surfaces only when the host keeps light-paper / dark-ink, but take the accent hue always. Legible on any theme, still on-brand. engineering →

◆ Verdict · /init validation

/init works — the harvest is the strong part; the template's assumptions are the gap. On a cold repo (`linkrot`) it read the source against the README and surfaced REAL defects (a directory EISDIR, a missing test) as the honest plan spine — not filler. But the template assumes a spec to anchor into, a count-deriving sidecar, and an id manifest — none exist in a stranger's repo, so I had to hand-patch them. Those assumptions ARE the packaging spec. the plan →

◆ Verdict · strategy

The browser is the eyes, not a worse mouth. Don't replicate the terminal's chat on the page — the terminal owns steering; the page's edge is state-made-visible and artifact-made-touchable. pivot →

◆ Verdict · architecture

claude -p is the same Claude, not a subpar one. Capability is a policy dial (--allowedTools), observability is in the stream; only interactive choices are staged. →

▲ Gotcha · concurrency

Optimistic direct file edits race unless the read-modify-write is atomic. Two quick Hands ops on async readFile→writeFile interleave — both read the old file, the second clobbers the first. Use sync read+write (no await points between), and refuse a direct write while an AI turn is editing the same page. the plan →

▲ Gotcha · CSS cascade

Extracting tokens to theme.css does nothing if the importer keeps its own :root. @import loads first, so a later duplicate :root in brain.css wins — overrides to theme.css silently no-op. The migrate step left it as "harmless redundancy"; it isn't. Drop the consumer's :root entirely. decision →

▲ Gotcha · amux deep-dive

One doc = one folder = one session; the file is the memory. So --resume is an accelerator, not a dependency — a fresh session rebuilds from the file. →

▲ Gotcha · fs.watch

fs.watch can fire with a null filename. Treat null as a match or you silently drop live-reloads (Linux/inotify, macOS under load). fix →

▲ Gotcha · env hygiene

Scrub the child env before spawning claude. An exported ANTHROPIC_API_KEY silently bills the API even on a plan; drop it + the CLAUDECODE markers. →
