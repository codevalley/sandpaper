# Sandpaper v0.3.0 First-Class Claude + Codex Design

**Status:** Approved for implementation on 2026-07-11

**Release:** `@nynb/sandpaper` `v0.3.0`

**Plan anchor:** `brain/project/index.html#i-providers`

## Purpose

`v0.3.0` makes Claude Code and Codex first-class Sandpaper providers. A normal installation configures both agent surfaces, the floating toolbar can deliberately select either provider, and the same provider-neutral brain workflows remain the durable project record.

The provider is replaceable compute, not a competing source of project truth. Repository files remain authoritative for behavior and rendered output. The brain remains authoritative for durable intent, decisions, plans, progress, and learnings. Provider sessions and browser transcripts are resumable convenience state only.

The implementation must preserve the `v0.2.1` safety and truth guarantees: the server owns the turn lifecycle, actual bytes determine Saved versus Replied, snapshots and Undo remain truthful, no provider silently replaces another, and published-package evidence remains a release requirement.

## Product decisions

The approved product contract is:

1. Use an internal provider registry with first-party Claude and Codex adapters. Do not expose a public provider-plugin API in `v0.3.0`.
2. Install both Claude and Codex integrations by default. The initial runtime provider is a separate local preference.
3. Store a local default provider in the gitignored `.sandpaper/manifest.json`. An explicit CLI choice determines the launch's initial provider; an explicit toolbar choice may then override it for that browser session.
4. Allow provider switching only while no turn is active. Never switch silently.
5. Keep page-scoped Claude and Codex sessions and transcripts separate. Returning to the same provider resumes its own conversation; crossing providers transfers no hidden context.
6. Expose an explicit, provider-scoped `New session` action. It clears only that page/provider's resume ID and browser transcript.
7. Install one Codex skill invoked as `$sandpaper <action>`, including `$sandpaper release`. Keep Claude's `/sandpaper:<action>` commands.
8. Reuse the user's saved Codex authentication, whether ChatGPT or an intentionally configured API-key login. Sandpaper injects no credentials and reports the detected method in `doctor`.
9. Run toolbar Codex turns in a controlled embedded configuration that ignores unrelated user/project rules and optional integrations. Interactive `$sandpaper` use continues to run in the user's normal Codex environment.
10. Match the current Claude operating boundary by running Codex from the real document directory. This is project-directory access, not hard single-file isolation, and must be documented honestly.
11. Keep provider-scoped toolbar history. A combined transcript is rejected because it would imply context that the newly selected provider never received.
12. Treat an unavailable provider as an explicit actionable state. Do not automatically fall back to another provider.

## Authority and durable state

Sandpaper has one provider-neutral project record with layered authority:

- Repository source files are the source of truth for implementation and rendered output.
- Brain HTML is the source of truth for product intent, decisions, plan state, worklog, and durable learnings.
- `.sandpaper/manifest.json` and `.sandpaper/session.json` are gitignored local runtime state.
- Claude and Codex native session IDs are opaque provider-owned resume handles.
- Browser transcripts are local convenience history and never override disk or brain state.

Claude, Codex, and a human may all change the same repository. Meaningful work is stamped into the same brain regardless of who performed it. Provider attribution may be recorded as historical evidence, but provider-specific session IDs, hidden reasoning, and authentication state must never enter the brain.

## Architecture

### Internal provider registry

Introduce an internal registry that is the only production path from provider identity to provider adapter. The first-party entries are `claude` and `codex`.

Each registry entry supplies:

- stable ID and user-facing label;
- availability and authentication diagnostics;
- a runner factory or `runTurn` implementation;
- native event normalization;
- provider-specific usage metadata;
- install and doctor metadata where needed.

The registry is internal and statically populated. `v0.3.0` does not scan npm packages, load arbitrary provider modules, or promise third-party API compatibility.

### Runner contract

Formalize the existing structural seam rather than changing the server lifecycle. Conceptually, an adapter receives:

```js
runTurn({
  pageFile,
  prompt,
  resumeId,
  onSession,
  onFrame
}) // returns an optional { kill() } handle
```

An adapter:

- invokes exactly one provider process per toolbar turn;
- reports a new/resumed native session ID through `onSession`;
- emits normalized status, assistant, edit, usage, warning, and terminal frames;
- emits or causes exactly one terminal outcome after provider filesystem activity settles;
- returns a killable process handle;
- never assigns Sandpaper turn IDs, chooses the active page, writes browser transcripts, decides Saved/Replied, owns snapshots, triggers reload, or exposes Undo.

The server remains authoritative for:

- one global active turn;
- provider and page validation;
- turn IDs and event replay;
- target hashes and actual-change truth;
- snapshot creation, pruning, and consumption;
- direct-edit conflict handling;
- terminal state and cleanup;
- page-scoped reloads.

### Provider identity in the protocol

Every accepted turn records its validated provider ID. The server attaches that ID to all frames and replay state. The browser-provided provider value is input, not authority; only registered and available providers can start a turn.

This permits separate browser tabs to select different providers while retaining the existing global one-turn-at-a-time invariant. A provider frame can never be appended to another provider's transcript merely because a tab changed its selection.

## Provider selection

Before the browser makes an explicit choice, initial provider resolution is:

1. explicit CLI `--provider claude|codex`;
2. local `.sandpaper/manifest.json` default;
3. migrated `claude` default for an existing pre-`v0.3.0` installation.

The CLI override controls the launch's initial choice and does not rewrite the manifest. After the toolbar loads, an explicit browser selection supersedes that initial choice for the browser session and survives reloads without changing the manifest. Only an explicit `Make default` action updates the local manifest.

If the selected provider is missing, unauthenticated, or incompatible, Sandpaper renders an unavailable state with exact repair guidance. The user may deliberately select another available provider. No automatic fallback is permitted because provider choice can change behavior, context, and billing.

## Sessions and transcripts

### Shared session store

Replace the current unqualified Claude-only `.sandpaper/session.json` shape with an atomically written versioned store, conceptually:

```json
{
  "version": 2,
  "pages": {
    "brain/index.html": {
      "claude": {
        "resumeId": "opaque-claude-session-id",
        "updatedAt": "2026-07-11T00:00:00.000Z"
      },
      "codex": {
        "resumeId": "opaque-codex-thread-id",
        "updatedAt": "2026-07-11T00:00:00.000Z"
      }
    }
  }
}
```

The exact field names may be refined during implementation, but these invariants are fixed:

- page and provider jointly scope every resume ID;
- a Claude ID can never be supplied to Codex or vice versa;
- malformed or unknown state fails closed for resume while leaving the document usable;
- updates are atomic;
- the legacy `{ "sessionId": "..." }` shape migrates to the current page's Claude entry without losing the ID;
- resetting one page/provider does not affect any other entry.

### Browser transcript storage

Transcript keys include the served project identity, page, and provider. Selecting Claude shows only Claude history; selecting Codex shows only Codex history. A provider switch does not clear either transcript. `New session` clears the selected transcript only after the server successfully clears the corresponding resume ID.

Browser history is not sent across providers. A future explicit handoff-summary feature may synthesize portable context, but cross-provider context transfer is outside `v0.3.0`.

## Provider runtimes

### Claude adapter

Preserve the existing `claude -p` behavior and stream mapping behind the formal adapter contract. Refactor only what is necessary to use the shared session store, shared startup-error rules, provider identity, and conformance tests.

Claude remains constrained to its current Read/Edit/Write/MultiEdit tool set. Existing users must not lose resume behavior, streamed prose/thinking, edit cards, or cost display when Claude supplies cost data.

### Codex adapter

Target a Codex CLI version that supports non-interactive JSONL execution, explicit resume IDs, `workspace-write`, non-interactive approvals, and the selected configuration overrides. `doctor` reports the detected version and produces an actionable incompatibility error when required capabilities are absent.

The intended initial invocation shape is:

```sh
codex \
  --ask-for-approval never \
  --sandbox workspace-write \
  --cd "$DOCUMENT_DIRECTORY" \
  --config 'web_search="disabled"' \
  --config 'sandbox_workspace_write.network_access=false' \
  --disable multi_agent \
  --disable apps \
  exec \
  --ignore-user-config \
  --ignore-rules \
  --json \
  "$PROMPT"
```

Resume uses the same controlled options with `exec ... resume "$THREAD_ID" "$PROMPT"`. Exact flag spelling and ordering must be exercised against the supported Codex version because global options must precede `exec` in current releases.

The toolbar runner:

- reuses saved Codex authentication;
- never passes credentials on the command line;
- removes `CODEX_API_KEY` and `OPENAI_API_KEY` environment overrides from the child;
- uses Codex's built-in default model rather than exposing a `v0.3.0` model selector;
- disables network access, web search, apps, and multi-agent behavior;
- ignores unrelated user configuration and repository agent rules;
- sets the real document directory as the writable workspace;
- instructs Codex to edit only the selected document.

Interactive Codex use of `$sandpaper` is not run through this controlled child process and continues to use the user's normal Codex configuration.

### Codex JSONL normalization

Map Codex's stable non-interactive lifecycle into Sandpaper's existing frame vocabulary:

| Codex event | Sandpaper behavior |
| --- | --- |
| `thread.started` | Persist the thread ID; emit/init provider state |
| `turn.started` | `thinking` status |
| command-execution item start | `tool_using` status |
| completed reasoning item | one complete thinking/assistant frame when available |
| completed agent-message item | one complete assistant frame |
| completed file-change item | edit summary using reported paths and kinds |
| todo-list item | optional progress frame; never lifecycle authority |
| item/top-level nonterminal error | visible warning while awaiting a terminal event |
| `turn.completed` | usage frame plus terminal success |
| `turn.failed` | terminal provider error |
| process exit without terminal event | synthesized terminal error |

Codex JSONL currently reports token usage rather than a dollar cost and reports file paths/kinds rather than Claude-style hunks. The UI must not invent missing cost, partial prose, diffs, or CIDs. Actual target bytes remain authoritative even when provider edit metadata is sparse or absent.

## Runtime safety and failures

The selected project-directory boundary intentionally matches the current Claude operating model more closely than an isolated staging copy. It is not a hard single-file guarantee.

Sandpaper therefore must:

- state the directory-level write boundary in runtime help and documentation;
- keep network and optional Codex integrations disabled for toolbar turns;
- make the target-only instruction explicit;
- surface provider-reported out-of-target file changes as a safety violation;
- never claim that Sandpaper can undo or verify changes outside the selected document;
- continue deriving Saved/Replied and Undo exclusively from the selected document's bytes;
- avoid automatically reverting unrelated paths, which could destroy concurrent human work.

Out-of-target detection is best-effort when a provider changes files through mechanisms not represented in its event stream. `v0.3.0` must not describe this as OS-level isolation. Strict staging or OS policy remains a possible later safety mode.

Startup and terminal errors become consistent across providers:

- an unavailable selected provider is rejected before turn reservation;
- a synchronous process-start failure clears the reservation and returns a structured start error;
- a process failure after start produces a terminal error frame;
- malformed events are ignored or surfaced as warnings unless lifecycle safety requires failure;
- EOF without an accepted terminal event becomes an explicit terminal error;
- partial target changes retain truthful change and Undo state.

## Toolbar experience

Replace the hard-coded Claude identity in the toolbar header with an accessible provider selector. Its menu contains:

- Claude Code with available/unavailable state;
- Codex with available/unavailable state;
- `Make default` for the current provider;
- `New session` for the current page/provider;
- provider-specific setup guidance when unavailable.

The selector is disabled while the global turn lifecycle is active. Switching while idle swaps the visible transcript and provider-specific composer label. It does not change the manifest or transfer context.

Provider identity appears in turn records, completion/error copy, accessibility labels, and replayed state. Welcome and Sling guidance become provider-aware or provider-neutral as appropriate.

Usage presentation accepts provider-specific data:

- display Claude dollar cost only when supplied;
- display Codex token usage when supplied;
- omit unavailable measures rather than estimating them.

The selector, menu, setup states, confirmation dialogs, and reset behavior must be keyboard-accessible and correctly named for assistive technology. Narrow-toolbar behavior is part of manual release qualification.

## Installation and canonical workflows

### Default dual installation

A normal `install-skill` configures both provider surfaces:

```text
.claude/commands/sandpaper/                  Claude slash-command wrappers
.claude/settings.json                        Claude hook wiring
CLAUDE.md                                    managed Sandpaper contract block

.agents/skills/sandpaper/SKILL.md            Codex dispatcher skill
.agents/skills/sandpaper/references/          shared workflow references
.codex/hooks.json                             Codex hook wiring
AGENTS.md                                     managed Sandpaper contract block
```

Sandpaper preserves user content and only owns explicitly delimited instruction blocks, its namespaced command/skill files, and its own hook entries. Hook merge and deduplication must be idempotent. Existing unrelated settings, hooks, instructions, and skills are never replaced wholesale.

Codex project hooks run only when Codex trusts the project. Installation output must explain that boundary instead of claiming the hooks are already active merely because configuration was written. `doctor` validates the installed configuration and reports trust as an external prerequisite when it cannot verify the current Codex trust decision.

An explicit integration option may install only Claude or Codex for constrained environments, but dual installation is the documented default. Runtime provider selection remains separate from integration installation.

### One workflow source

Move durable action semantics into provider-neutral workflow documents. Provider-specific entrypoints are thin adapters:

- Claude keeps `/sandpaper:init`, `/sandpaper:plan`, `/sandpaper:release`, and the existing namespace.
- Codex installs one discoverable `$sandpaper` skill and dispatches `$sandpaper init`, `$sandpaper plan`, `$sandpaper release`, and equivalent actions.
- Claude-only constructs such as `$ARGUMENTS`, command frontmatter, or `AskUserQuestion`, and Codex-only skill routing stay outside the canonical workflow body.

Both entrypoints read and stamp the same brain. Neither provider receives a separate plan, worklog, decision ledger, or release process.

### Migration and lifecycle commands

- Fresh `init` asks for the initial provider when interactive and accepts an explicit non-interactive value.
- Existing Claude-only manifests migrate to default `claude` without changing behavior.
- `upgrade` refreshes installed integrations, managed blocks, and hook scripts without clobbering user content.
- `rebuild` preserves provider default, installed-integration selection, `--no-hooks` intent, brain identity, and ID counters.
- Repeated install, upgrade, and rebuild runs converge to one valid configuration.

### Doctor

Expand `doctor` to validate and clearly classify:

- manifest and session schema versions;
- selected provider and local default;
- Claude and Codex binary presence and compatible capabilities;
- Codex saved authentication method;
- installed Claude commands and Codex skill/reference files;
- managed `CLAUDE.md` and `AGENTS.md` blocks;
- provider-specific hook configuration and shared hook scripts;
- duplicates, stale generated files, and migration residue;
- published-package identity and expected shipped files.

An unavailable optional provider is a warning. An unavailable selected provider, corrupt selected-provider configuration, or unsafe/invalid hook merge is an error with an exact repair command.

## Documentation and packaging

Update current-state surfaces to describe both providers accurately:

- README and CLI help;
- package description and keywords;
- install, init, upgrade, rebuild, doctor, and next-step output;
- public landing page;
- product and engineering specifications;
- current brain cover, map, product, engineering, project, wiki, decisions, learnings, digest, and worklog where the current state changes;
- toolbar welcome, composer, Sling, status, error, and accessibility copy.

Do not rewrite historical evidence that was truthfully Claude-specific at the time. New documentation shows `/sandpaper:<action>` and `$sandpaper <action>` together and states the controlled Codex runtime, authentication reuse, token-versus-cost difference, directory-level write boundary, and lack of cross-provider hidden-context transfer.

Add every new runtime, skill, workflow, fixture, and documentation asset intentionally to `package.json.files`, `verify-publish`, and the exact package-smoke allowlist. Re-evaluate the package size envelope from the intentional artifact; do not weaken path or secret exclusions merely to fit new files.

## Testing

CI never calls live Claude or Codex. Automated tests use recorded streams, injected process fakes, disposable repositories, real local HTTP servers, and Chromium.

### Adapter and server tests

- shared runner conformance for terminal behavior, kill handles, sessions, errors, and normalized frames;
- recorded Claude and Codex success, reply-only, edit, failure, malformed-event, unexpected-EOF, and interruption streams;
- Codex invocation flags, ordering, cwd, environment scrubbing, resume ID, and capability diagnostics;
- Claude regression coverage for invocation, resume, environment, streaming, cost, and startup errors;
- provider validation, identity attachment, replay, global concurrency, target-byte truth, partial-change errors, snapshots, and Undo;
- versioned session migration, atomic persistence, page/provider scoping, reset, and corrupt-state recovery.

### Setup and package tests

- fresh Claude-only, Codex-only, and dual-integration fixtures;
- pre-existing `CLAUDE.md`, `AGENTS.md`, settings, hooks, skills, and commands;
- idempotent merge/deduplication;
- upgrade and rebuild preference preservation;
- `--no-hooks` behavior;
- selected, missing, unauthenticated, and incompatible providers in `doctor`;
- packed-package install followed by help, install-skill, init, doctor, open, and fake provider turns;
- exact tarball paths, secret scan, package identity, and intentional size envelope.

### Browser tests

- provider metadata injection and accessible selector rendering;
- available/unavailable states and no silent fallback;
- browser-session override and explicit `Make default`;
- provider-scoped transcripts, switching, reload, reset, and resume;
- provider-labeled frames and protection against cross-provider replay contamination;
- selector locking during a turn and global busy behavior across tabs;
- Claude cost versus Codex token usage;
- provider-specific copy, errors, keyboard behavior, focus, screen-reader labels, and narrow layout;
- all existing `v0.2.1` toolbar, direct-edit, rollback, Undo, reload, and CSS-isolation regressions.

## Release contract

`v0.3.0` is a minor release because it adds an official provider, new install surfaces, new runtime selection, and new public command syntax while preserving existing Claude behavior.

Release qualification requires:

1. All shipped-JavaScript syntax checks.
2. Node unit/integration suites across supported Node versions.
3. Chromium browser suite.
4. `doctor` on the repository and representative Claude-only, Codex-only, dual, upgraded, and broken fixtures.
5. `verify-publish`, exact tarball inspection, and packed-package installation.
6. A disposable real Claude edit followed by resume.
7. A disposable real Codex edit followed by resume.
8. A real provider switch proving independent continuity and provider-scoped reset.
9. Missing-provider recovery and authentication-method verification.
10. Manual desktop and narrow-toolbar review.
11. Brain, README, specifications, site, package metadata, and changelog stamped to the same current state.

The release is blocked if either provider's core installation, edit, resume, switch, reset, or package-smoke path fails. Use the provider-neutral release workflow through `/sandpaper:release` or `$sandpaper release`; a human confirms the semver bump and tag. The tag-triggered workflow re-verifies, publishes with provenance, and creates the GitHub Release.

Completion requires registry version `0.3.0`, provenance, a successful fresh `npx @nynb/sandpaper@0.3.0 help`, working clean installation, and a GitHub Release. Published artifacts remain immutable.

## Future provider plugin path

The internal registry deliberately creates a future seam without creating a current public API. A later plugin system could allow packages such as `@sandpaper/provider-example` to supply runners, diagnostics, metadata, and install surfaces.

Before that seam can become public, Sandpaper must define:

- a versioned provider manifest and compatibility policy;
- package discovery and explicit trust/enablement;
- process and filesystem security requirements;
- event-schema stability and conformance certification;
- failure isolation and support boundaries;
- packaging, documentation, and upgrade rules for third-party providers.

No dynamic provider loading, third-party package execution, or compatibility promise ships in `v0.3.0`.

Implementation and compatibility checks must prefer current official documentation and the supported CLI's own `--help` output:

- [Codex non-interactive execution and resume](https://developers.openai.com/codex/noninteractive)
- [Codex authentication](https://developers.openai.com/codex/auth)
- [Codex CLI reference](https://developers.openai.com/codex/cli/reference)

Codex CLI behavior is versioned. Recorded fixtures and capability probes, rather than undocumented assumptions, define the supported adapter contract.

## Non-goals

`v0.3.0` does not include:

- seamless Claude-to-Codex hidden-context transfer;
- generated cross-provider handoff summaries;
- per-prompt provider selection UX;
- a public provider plugin API;
- third-party provider loading;
- a Codex model selector;
- persistent provider processes;
- hard single-file OS isolation or staging mode;
- automatic provider fallback;
- unrelated Phase 0/Phase 1 open features.

## Success criteria

- A fresh default install configures both providers without clobbering existing agent files.
- Claude slash commands and the Codex `$sandpaper` skill execute the same durable workflows.
- The toolbar can deliberately select either available provider and identifies every turn correctly.
- Provider sessions and transcripts are page-scoped, independent, resumable, and individually resettable.
- Cross-provider switches transfer no hidden context and never pass an incompatible resume ID.
- Codex uses saved authentication in a controlled, offline, project-directory toolbar runtime.
- Missing providers and failures are explicit; no silent fallback or false Saved state occurs.
- The server retains lifecycle, byte-truth, snapshot, Undo, and reload authority.
- `doctor`, tests, package smoke, documentation, brain state, and release evidence all describe the same shipped artifact.
- Claude behavior remains backwards-compatible while Codex is functionally and documentationally first-class.
