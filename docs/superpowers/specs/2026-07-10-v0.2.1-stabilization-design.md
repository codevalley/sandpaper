# Sandpaper v0.2.1 Stabilization Design

**Status:** Approved for implementation on 2026-07-10

**Release:** `@nynb/sandpaper` `v0.2.1`

**Plan anchor:** `brain/project/index.html#i-stability`

## Purpose

`v0.2.1` is a truth-and-safety patch release before first-class Codex support. It fixes verified runtime, browser, setup, documentation, and release defects without adding unrelated roadmap features. The document on disk remains the single source of truth, the runtime remains bound to `127.0.0.1`, and the published package keeps zero runtime dependencies.

The release is complete only when the implementation, regression coverage, brain stamp, package metadata, changelog, publish verification, and fresh-install checks all agree.

## Scope

The release includes:

1. A shared repository path policy that preserves Sandpaper's link-never-copy presentation while preventing secret and runtime-file disclosure.
2. A server-owned turn lifecycle that removes request races and makes SSE, reload, snapshot, error, and undo state truthful.
3. A hardened browser API and repaired Pick, Hands, direct-edit, undo, search, styling, keyboard, and accessibility behavior.
4. A reconciled brain and documentation layer in which mechanical facts are derived and editorial judgment remains explicitly stamped.
5. Node integration and Chromium browser regression coverage, plus stricter release gates.

The release does not include first-class Codex execution, provider selection, screenshot paste, branching, durable regenerated-element anchoring, keyboard/touch structural editing, or other open Phase 0/Phase 1 roadmap work. Claude remains the only execution provider and is named honestly in the `v0.2.1` UI and documentation. The internal runner seam is provider-neutral so `v0.3.0` can add Codex without another server rewrite.

## Architecture

### Repository boundary

Add one shared path-policy module used by both the HTTP server and `doctor`. It classifies a repository-relative path before it is served or accepted as a brain reference.

- Ordinary repository files remain readable so links to specs, source, assets, package files, and documentation continue to work.
- `.github/**` is the only built-in safe hidden namespace because the current brain deliberately links to release workflows there.
- `.git/**`, `.sandpaper/**`, unknown hidden namespaces, environment files, npm/netrc credentials, private keys, certificates, and credential-shaped files are denied.
- Percent-encoded traversal, lexical traversal, null bytes, and symlinks resolving outside the served root are denied.
- Hidden paths are never valid direct-edit or turn targets.
- `doctor` evaluates local `href`, `src`, and `data-ref` values with the same policy. A future link into `.codex`, `.claude`, or another hidden namespace therefore fails during stamping instead of becoming an invisible production break.

The server still serves the requested document folder in single-document mode and the repository root in whole-repo brain mode. Removing repo-root serving is out of scope because it would break the product's canonical-link model.

### Local trust boundary

Each server process generates an in-memory cryptographically random token and injects it into the toolbar bootstrap at response time; it is never written to the document or repository. Browser requests use one centralized API helper.

- Mutation endpoints require JSON, the process token, a valid local host, and the expected same-origin `Origin`.
- SSE receives the token through its connection URL because `EventSource` cannot set custom headers.
- Static document requests remain directly navigable but are constrained by the repository policy.
- Invalid authentication, origin, content type, body size, JSON, page, CID, operation, or snapshot identifiers return structured JSON errors with stable HTTP status codes.
- The browser always restores its interactive state after a rejected request and presents the server's actual error.

This is local-process protection against cross-origin browser requests, DNS rebinding-style access, and accidental secret serving; it is not a multi-user authentication system.

### Turn lifecycle and runner seam

Refactor `src/server.js` around a testable `createSandpaperServer(...)` construction API while preserving the existing `startServer(...)` CLI contract. The constructor accepts an injected runner; production defaults to the Claude runner, while tests use a deterministic fake runner.

The server owns one lifecycle:

`idle -> receiving -> running -> done | error -> idle`

- A turn reserves the lifecycle before reading its request body, so two simultaneous uploads cannot both start.
- The server stores and replays the current status to newly connected SSE clients.
- The runner only emits normalized frames. The server owns turn IDs, page association, snapshots, hashes, locking, terminal state, reloads, and cleanup.
- The server hashes the target file before and after the runner finishes. `Saved` and AI undo are exposed only when bytes actually changed; a reply without a disk change is `Replied`.
- A terminal error remains visible and is not immediately overwritten by an idle frame. A later accepted turn begins the next lifecycle.
- Direct edits and undo are rejected while an AI turn can conflict with the same page.
- Snapshots are created only when useful, consumed after undo, and pruned to a fixed bounded retention limit. A failed or unchanged operation does not expose a false undo.
- The file watcher identifies the initiating browser by a per-tab client ID. The initiating tab may retain its optimistic direct edit while other tabs receive the page-scoped reload. AI changes and external changes reload every relevant tab after the active reply is safe to finish.

The fake runner must deterministically simulate reply-only, actual edit, attempted-but-unchanged edit, delayed execution, malformed frames, and process failure.

### Browser contract

`public/toolbar.js` uses one API helper for tokens, client IDs, JSON encoding, structured errors, and final UI restoration. It does not infer success from HTTP acceptance or model tool requests.

- Pick and Hands are mutually exclusive modes. Mode buttons expose `aria-pressed`; the panel exposes its expanded state.
- Composer submission disables only while needed and always recovers after 4xx, 5xx, malformed, network, or SSE terminal errors.
- The completion card says `Saved` only when the server reports a real file change and supplies an undoable turn ID. Otherwise it says `Replied` and shows no undo affordance.
- Direct text, move, and delete operations remain optimistic, but each captures the exact original DOM state and restores it on a rejected write.
- AI and direct undo are rejected during an active turn, shown only for a real snapshot, and removed after consumption.
- Intentional rich `innerHTML` editing remains supported. It is safe because the human is editing a trusted local document; sanitizing it would remove a core feature.
- The toolbar's delete control no longer shares `.sp-del` with rendered document content. Toolbar styles are scoped to the Sandpaper overlay and explicit page-state hooks.
- Brain timeline filtering actually hides unmatched rows.
- `/` search focus and other shortcuts ignore inputs, textareas, selects, buttons, and contenteditable regions.
- Status and transcript updates use appropriate live-region roles; icon controls have accessible names; welcome behavior contains and restores focus; motion obeys `prefers-reduced-motion`.

Keyboard/touch reorder and delete remain tracked as later task `t-0024` rather than being silently folded into this patch.

### Brain and documentation truth

The existing HTML entries remain canonical. No plan JSON database or new documentation store is introduced.

`brain/assets/brain.js` derives mechanical presentation values from the same-origin HTML books and plan:

- task and initiative progress;
- open-question count and list presentation;
- decision and learning counts;
- component/build rollups that can be computed from explicit entry state.

Stamped values remain in the HTML as a readable `file://` and no-JavaScript fallback. `doctor` independently recomputes and validates those fallbacks so derivation does not hide drift.

Editorial state remains stamped: NOW, digest focus, newest worklog entries, decisions, learnings, and the curated open list. `doctor` checks digest JSON, focus references, newest-log agreement, open-question statuses, local references under the shared path policy, source metadata consistency, and package identity. The repo brain's `data-pkg` must be `@nynb/sandpaper`, matching `package.json`.

Perform a one-time truth sweep across the cover, map, plan, README, wiki, product spec, and engineering spec. Resolve the obsolete undo question, report Hands and shipped capabilities accurately, regenerate derived engineering documentation through the existing workflow, and preserve historical worklog and decision entries unchanged. Provider-neutral terminology may appear only at the runner seam; user-facing current-state documentation remains explicit that `v0.2.1` runs Claude.

## Testing

The runtime keeps zero dependencies. Playwright is a development-only dependency and is excluded from the published `files` allowlist.

### Node tests

Run the existing parser, markdown, and edit tests plus new unit/integration suites on Node 18, 20, and 22. Use real HTTP servers on ephemeral ports and temporary repositories; inject the fake runner rather than launching an agent.

Coverage includes:

- allow ordinary repository files and `.github`, deny secrets, runtime state, unknown hidden namespaces, traversal, and escaping symlinks;
- authentication, origin, host, content type, body-size, malformed JSON, and structured errors;
- concurrent turns with exactly one accepted request;
- current-state SSE replay and terminal error retention;
- reply-only versus actual-change completion;
- AI and direct undo, missing/consumed snapshots, no-op writes, undo during a turn, and retention pruning;
- write, move, and delete validation;
- client/page-scoped reload behavior;
- setup, install, upgrade, rebuild, source metadata, doctor policy, metadata mismatch, brain derivation, and fallback validation.

### Browser tests

Run Playwright Chromium once on Node 20 against the real server with the fake runner. Cover:

- toolbar injection and rendering;
- `Saved` versus `Replied` and undo visibility;
- 409, authentication, malformed, network, and runner-error recovery;
- Pick/Hands exclusivity;
- direct text, move, delete, exact rollback, and undo;
- two-tab reload behavior;
- timeline filtering and shortcut guards;
- ARIA state, focus management, reduced motion, and host-page CSS isolation.

CI never calls live Claude or Codex. Before release, run one manual Claude turn against a disposable fixture and record the result in the brain.

## Release contract

`v0.2.1` remains a patch release because it corrects safety, truth, and reliability without advertising a new provider or unrelated product surface.

Release gates are:

1. Syntax-check every shipped JavaScript file, including `bin/`, `src/`, `public/`, and `brain/assets/`.
2. Run Node tests on 18, 20, and 22.
3. Run Playwright Chromium on Node 20.
4. Run `sandpaper doctor` against the current repo and representative clean fixtures.
5. Run `npm run verify-publish` and inspect the packed file list.
6. Install the packed package into a fresh temporary repository and exercise `help`, `install-skill`, `init`, `doctor`, and `open`.
7. Run and record one disposable live Claude edit.
8. Populate `CHANGELOG.md` from the stamped brain, including the post-`v0.2.0` awk release-note correction and the stabilization sweep.

Use `/sandpaper:release` for the final patch proposal. A human confirms the version bump and push. The tag-triggered GitHub workflow runs the gates again, publishes with provenance, and creates the GitHub Release.

Release is not done until the registry reports `0.2.1`, provenance is present, `npx @nynb/sandpaper@0.2.1 help` succeeds, and a fresh installation works. Published tags and packages are immutable; a release defect is corrected with a new patch version rather than rewriting `v0.2.1`.

## Success criteria

- No repository secret or runtime namespace is served by default, while all intentional current brain references—including `.github`—continue to resolve.
- Concurrent requests cannot start multiple agent turns.
- Every browser-visible state corresponds to server-confirmed state and real disk bytes.
- Direct edits either persist or restore their exact prior DOM state.
- Brain mechanical facts and stamped fallbacks agree, and `doctor` detects divergence.
- Automated tests cover the verified defects without live-agent CI calls.
- The manual tester receives a changelog-derived checklist that maps each release claim to a reproducible action.
