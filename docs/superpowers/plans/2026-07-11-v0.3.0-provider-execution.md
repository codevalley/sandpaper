# Sandpaper v0.3.0 Provider Execution Map

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship first-class Claude Code and Codex support without regressing Sandpaper's server-owned truth, install safety, toolbar recovery, or release evidence.

**Architecture:** Build and review one dependency chain: provider runtime primitives first, provider-aware distribution second, toolbar and release surfaces last. Fresh implementation subagents execute one task at a time; task-scoped reviewers gate every commit, while read-only audits may run in parallel.

**Tech Stack:** Node.js 18+ ESM with zero runtime dependencies, built-in HTTP/SSE/filesystem/process APIs, Claude Code stream-json, Codex `exec --json`, static HTML/CSS/JavaScript, Node test runner, and Playwright Chromium.

## Global Constraints

- Repository files remain authoritative for behavior and rendered output; brain HTML remains authoritative for durable intent, decisions, plan state, and learnings.
- The server continues to own global turn concurrency, validated provider identity, target hashes, snapshots, Saved/Replied, Undo, terminal cleanup, and reload.
- Install both Claude and Codex integrations by default; preserve explicit Claude-only or Codex-only installations.
- Never silently fall back between providers.
- Page/provider sessions and transcripts remain independent; crossing providers transfers no hidden context.
- Toolbar Codex turns reuse saved authentication, remove API-key environment overrides, ignore external rules/configuration, and disable network, web search, apps, and multi-agent execution.
- Codex runs with `workspace-write` from the real document directory; documentation must not claim hard single-file isolation.
- Preserve zero runtime dependencies and Node.js `>=18`.
- CI uses fakes and recorded streams only; live Claude and Codex turns are manual release gates.
- A public provider plugin API, cross-provider handoff summaries, per-prompt provider selection, model selection, persistent provider processes, strict staging, and unrelated roadmap features are out of scope.

## Shared Interfaces

All linked plans use these names and shapes:

```js
createProviderRegistry(entries)
// => { get(id), diagnostics() }

createProviderPreferenceStore(root)
// => { getDefaultProvider(), setDefaultProvider(id) }

createSessionStore(root)
// => {
//   get({ page, provider }),
//   set({ page, provider, resumeId }),
//   clear({ page, provider }),
//   inspect()
// }

providerEntry
// => {
//   id,
//   label,
//   diagnose(),
//   runTurn({ pageFile, prompt, resumeId, onSession, onFrame })
// }
```

The browser/server protocol uses:

```text
data-sandpaper-bootstrap
POST /__sandpaper/turn
POST /__sandpaper/provider-default
POST /__sandpaper/session/reset
```

Every accepted turn, normalized frame, and replay record carries the server-validated `provider`.

## Execution Discipline

For every implementation task:

1. Extract only that task into a task brief.
2. Record the pre-task commit.
3. Dispatch one fresh implementation subagent using test-driven development.
4. Require a self-review, focused test evidence, and one or more task commits.
5. Generate a diff review package from the recorded base to the task head.
6. Dispatch a fresh task reviewer for both specification compliance and code quality.
7. Fix and re-review every Critical or Important finding before starting the next task.
8. Append the completed task and reviewed commit range to `.superpowers/sdd/progress.md`.

Do not run multiple implementation subagents concurrently in this shared worktree. Independent read-only research, fixture analysis, and final review preparation may run in parallel.

## Final Automated Gate

Run from the repository root:

```bash
npm run check:syntax
npm run test:unit
npm run test:browser
npm run test:package
node bin/cli.js doctor
npm run verify-publish
git diff --check
git status --short
```

Expected result: every command exits zero; `doctor` reports healthy; the final status contains only intentional, reviewed release work before its commit and is clean after the release-candidate commit.
