# Sandpaper v0.2.1 Stabilization Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Deliver a manually testable `v0.2.1` release candidate that closes the verified safety, lifecycle, browser-state, brain-truth, and release-coverage gaps without adding Codex or unrelated roadmap features.

**Architecture:** A shared repository path policy protects both HTTP serving and brain references. A testable server controller owns authenticated requests, one turn state machine, snapshots, hashes, SSE, and reload attribution around an injected provider runner. The browser consumes that contract through one client helper, while the existing HTML brain remains canonical and derives only mechanical presentation values.

**Tech Stack:** Node.js 18+ ESM and built-ins; static HTML/CSS/JavaScript; `node:test`; Playwright `1.61.1` as a development-only dependency; GitHub Actions.

## Global Constraints

- Keep zero runtime dependencies; `@playwright/test` is development-only and must not enter the published package.
- Bind HTTP only to `127.0.0.1`; accept browser API requests only from the same loopback origin and process token.
- The document on disk remains the single source of truth; never build document HTML from agent stdout.
- Preserve repo-root serving and intentional `.github/**` references; deny secrets, `.sandpaper/**`, `.git/**`, unknown hidden namespaces, traversal, and escaping symlinks.
- Preserve the public `startServer(target, port, opts) -> Promise<string>` behavior used by `bin/cli.js`.
- Keep `v0.2.1` user-facing language Claude-only. Provider-neutral naming is limited to the internal runner seam.
- Preserve historical decision and worklog entries. Resolve status and update current-state surfaces without rewriting history.
- Do not implement Codex, provider selection, screenshot paste, branching, regenerated-element anchoring, publish-preflight prose scanning, or keyboard/touch structural editing.
- Follow red-green-refactor for every behavior change: run the focused test and observe the expected failure before changing production code.
